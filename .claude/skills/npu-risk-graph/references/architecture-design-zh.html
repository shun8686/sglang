NPU 风险知识图谱 架构设计文档 · v2.0 · 2026年6月

摘要： NPU 风险知识图谱（npu-risk-graph）是一个多层分析系统， 从异构数据源构建属性图，计算风险评分、检测盲点、推荐 Oracle 测试模式， 并为 SGLang Ascend NPU 后端的测试投资决策提供指导。 v2.0 新增 测试知识系统：4 种 Oracle 模式、7 种故障模式、自动检测扫描器、 图谱驱动的测试设计推荐。

核心设计原则

  1. 单一数据源：features.json 是唯一权威特性注册表。
  2. 确定性优先，Agent 兜底：机械匹配处理常见情况；Agent 语义范围限定处理边界场景。
  3. 默认为增量模式：Delta 引擎仅检测变更文件，仅重新评分受影响特性。
  4. 图谱驱动测试设计：缺陷画像 → 推荐 Oracle 模式 → 生成测试代码。

3. 测试知识系统 NEW in v2.0

核心问题：如何判断一个测试"好"还是"不好"？ 答案取决于它用了什么 Oracle 模式——即测试判断"通过/失败"的策略。 四种模式按严格程度排列，对应不同的缺陷检测能力。

3.1 四种 Oracle 模式

模式严格度怎么做安全距离能检测盲区
gsm8k_accuracy 最弱 跑 1319 道 GSM8K 数学题，看正确率 ≥ 阈值 不限 crash、灾难性故障 静默精度丢失、token 偏离、accept rate 错误
exact_match_short 硬 Oracle 双服务器 temperature=0，逐字比对 16 token 16 tokens crash、显式 token 错误、逻辑 bug bfloat16 平局翻转（>16 tokens）、不改变 argmax 的 logit 偏差
logprob_rescore 软 Oracle 单服务器 decode vs prefill 重评分，logprob 差值 < 0.255 64 tokens 静默精度丢失、logit 错误、crash、token 偏离 性能回退
token_oracle_canary 最强（仅 CUDA） 确定性哈希采样器——任意 token 偏差立即报错 不限 一切——包括位置错误、KV cache 损坏 NPU 不支持（2026-06 验证失败）

bfloat16 精度限制（exact_match_short 为何限 16 token）

这是本系统通过实战验证发现的测试知识：

  • 根因：投机解码的 batched forward 与 sequential forward 在 bfloat16 下浮点累加顺序不同，差异约 1e-4 级别。
  • 触发条件：当两个 token 的 logprob 差异 < 1e-4 时（即精确平局），argmax 可能翻转。
  • 实证：eagle3 Oracle 测试中，baseline 在 token 14648 和 78158 上 logprob 均为 -1.250591（精确平局）。Eagle3 因 bfloat16 噪声打破平局，分别变为 -1.286224 和 -1.161224。
  • 结论：精确比对在 16 token 内几乎不会遇到平局；长序列需改用 logprob_rescore + tolerance 0.255。

3.2 故障模式分类（7 种）

故障模式静默?示例
crash否服务器崩溃、CUDA/NPU kernel 故障
silent_precision_loss是输出看似正确但数值有偏差——位置编码错误、logit 偏移
token_divergence否temperature=0 下 token 序列与参考不一致
accept_rate_error是投机接受率算错，target model 可纠正但吞吐受损
logit_computation_error是tree attention 掩码错误或 KV cache 损坏导致 logit 偏差
performance_regression是吞吐/延迟退化，不影响正确性
compatibility_issue否模型或配置兼容性破坏

3.4 自动检测扫描器

detect_oracle_patterns.py 通过 AST 分析自动从现有测试代码中识别 Oracle 模式：

Oracle 模式检测信号检测精度召回率
gsm8k_accuracy继承 GSM8KMixin / GSM8KAscendMixin100%100%
exact_match_shortassertEqual(output, ref) + temperature=0 + max_new_tokens ≤ 32 + 生成调用75%75%
logprob_rescoretest_logprob_* 方法 + max_new_tokens=0 重评分阶段 + tolerance 断言93%88%
token_oracle_canary继承 CanaryE2EBase + ENABLE_TOKEN_ORACLE 环境变量100%78%
总体（加权）0.75 置信度阈值过滤97%92%

扫描覆盖 823 个测试文件，识别出 98 个高置信度 Oracle 模式关联（全部自动入库）、14 个低置信度候选（待人工审核）。

3.5 NPU 图谱覆盖现状

NPU 图谱中 39 个测试的 Oracle 模式分布：

  • gsm8k_accuracy：39/39（100%）——全部依赖最弱的 Oracle
  • exact_match_short：0
  • logprob_rescore：0
  • token_oracle_canary：0（2026-06 验证：NPU 不支持，CUDA only）

→ NPU 测试栈对静默精度丢失零检测能力。 可用 Oracle 仅 3 种：gsm8k_accuracy、exact_match_short、logprob_rescore。 attention_backend（risk=20，有 silent_precision_loss 缺陷）零 Oracle 覆盖。

4. 风险评分模型

与 v1.0 相同，保持不变。每个特性的综合评分 = 概率因子（5项，最高25分）+ 影响因子（2项，最高20分）= 最高45分。

因子评分来源
code_complexity1-5AST 圈复杂度估算
change_frequency1-5Git：365天内的时间衰减加权提交数
historical_defects1-5defect_db：历史缺陷计数
dependency_depth1-5features.json：依赖特性数
test_effectiveness1-5测试评估器：最佳测试质量 0-5 → 反比映射
user_exposure1-5人工标注：使用的模型族数量
failure_mode1-5人工标注：故障后果严重度

风险等级：critical (≥28) → high (20-27) → medium (12-19) → low (<12)。

6. 知识图谱相较于纯 JSON 的增量价值

补充 v1.0 的 5 项价值，v2.0 新增：

  • Oracle 模式盲区：图谱可以回答"test_npu_eagle3 能检测静默精度丢失吗？" — 沿 TEST_USES_PATTERN → PATTERN_BLIND_SPOT 遍历即可得到答案。纯 JSON 需要人工理解两种数据并推理。
  • 缺陷驱动的 Oracle 推荐：图谱根据 feature 的历史缺陷 failure_mode，反向查 PATTERN_DETECTS 边， 自动推荐能覆盖这些故障的 Oracle 模式组合。这是多跳推理，JSON 无法做到。
  • 测试投资全景：--pattern-coverage 一目了然地展示哪些特性零 Oracle 覆盖、 哪些 Oracle 模式完全未被使用。驱动测试投资的全局优先级排序。
