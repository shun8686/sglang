npu-defect-mine — 设计文档

NPU 缺陷数据库全生命周期管理：PR-first 提取 → Agent 语义分类 → 数据完整性守护 → 周期性审计

🎯 概览

技能目标

为 sglang 项目的 NPU（华为昇腾 Ascend）相关代码缺陷建立结构化知识库。 从 GitHub 合并的 PR 中自动提取缺陷数据，由 AI Agent 进行语义分类， 并通过周期性审计确保数据质量。服务于下游的 NPU 风险测试框架。

🏁 init — 首次建库

6 步（A-F）：PR 提取 → 写入 DB → Agent 分类 → 合并 → 全量审计 → 一致性检查。 从 2025-01-01 起扫描所有已合并 NPU 缺陷 PR。PR body 在提取阶段嵌入，100% 覆盖。

📥 update — 增量更新

4 步（A-D）：扫描新 PR → Agent 分类 → 合并 → 可选一致性检查。 每次 ~0-5 条新缺陷。PR-first only，commit fallback 已移除。

🔧 maintain — 质量维护

5 步（A-E）：完整性检查 → Agent Audit → 应用修正 → 删除非 NPU → 审计计数 + 重检。 日常抽样 ~30 条，支持 --full 全量审计。adversarial verify 确保分类质量。

📊 status — 状态查看

只读查询，输出总量、needs_review 数量、category 分布、confidence 分桶、agent_version 分布。

🔄 子工作流详解

Sub-workflow 1: init — 首次建库

  1. Step A: PR 提取 — extract_prs.py --since 2025-01-01 → pr_defects.json（含 body/labels/reviews/files/merge_commit）
  2. Step B: 写入 DB + 批次准备 — seed_defect_db.py --from-prs --agent → defect_db.json + pending_agent_batch.json
  3. Step C: Agent 语义分类 — Claude 读取 pending_agent_batch.json，调用 Workflow 工具进行 pipeline(pr_batches, classify, verify)
  4. Step D: 合并结果 — apply_agent_results.py 将 Agent 产出写入 defect_db.json
  5. Step E: Agent Audit — audit_sample.py --all --split 15，并行 spawn Audit Agent → 主 Agent 分组确认 → 应用修正
  6. Step F (可选): 一致性检查 — consistency_check.py

Sub-workflow 2: update — 增量更新

  1. Step A: 扫描 + 写入 + 批次 — append_daily.py --agent，gh search prs merged:>={last_scan_at} + label:npu/ascend，四维校验后写入 DB
  2. Step B: Agent 分类 — 同 init Step C
  3. Step C: 合并 — apply_agent_results.py
  4. Step D (可选): 一致性检查

Sub-workflow 3: maintain — 质量维护

  1. Step A: 完整性检查 — consistency_check.py（必填字段、格式、枚举、值域、孤儿 commit）
  2. Step B: Agent Audit — 日常抽样 ~30 条（5 categories × 2 bands × 3 + compatibility 双倍 + needs_review），并行 spawn Audit Agent（7-8 个）→ 主 Agent 分组确认（A/B/C/D 四组）
  3. Step C: 应用修正 — apply_agent_results.py --input audit_report.json
  4. Step D: 删除非 NPU 缺陷 — 主 Agent 从 DB 中移除 action=delete 的条目
  5. Step E: 审计计数 + 重检 — audit_finalize.py + consistency_check.py

🔍 PR 提取引擎

提取流程

  1. Label 搜索 — gh search prs label:npu + label:ascend（去重）
  2. 关键词标题搜索 — "NPU"+"fix"、"Ascend"+"fix"、"CANN"+"fix" 等（补充无标签 PR）
  3. Fallback: git log + PR 关联 — 对无 PR 的 commit，通过 gh search prs sha:{} 反查
  4. 获取完整数据 — PR body、files、labels、reviews、merge_commit，并执行 is_genuinely_npu_defect() 校验

四维 NPU 校验 — is_genuinely_npu_defect()

每条 PR 必须满足以下至少一项才算真正的 NPU 缺陷：

维度检查内容
1. Labellabels 中含 "npu" 或 "ascend"
2. Title tag标题含 [NPU] / [Ascend] / [npu] / [ascend]
3. File path文件路径含 "npu" 或 "ascend"（如 hardware_backend/npu/）
4. Body discussionPR 正文中含实质性 NPU 讨论（剥离 Accuracy/Speed Tests 样板后匹配）
⚠ 提取过滤器（在源端跳过）： Revert PR（title 以 "Revert" 开头）· 纯文档（所有文件在 docs/ 或 .md/.rst/.txt）· 非产品代码（无文件在 python/、sgl-kernel/、sgl-router/）

Word-Boundary 正则（误报预防）

# ✅ 正确 — 使用 \b 词边界
\bnpu\b    # 匹配 "npu" 但不匹配 "input" (i-n-p-u-t), "nonpublic", "unplug"
\bcann\b   # 匹配 "cann" 但不匹配 "cannot", "scanning", "scanner"
\bascend\b # 不匹配样板启动命令中的 "ascend"

# 已知误报来源：
# "--device npu --attention-backend ascend" 在 PR 的 "Accuracy Tests" 章节中
# 是样板文本，不代表实质性 NPU 讨论。已在 body 检查中剥离。

🤖 Agent 语义分类

AGENT_PR_PROMPT

输入：PR title、body（2000 chars）、labels、review comments、linked issues、files changed

置信度：0.80-0.95（PR 数据丰富）

使用场景：所有 source=pr 的缺陷

AGENT_COMMIT_PROMPT

输入：commit SHA、message、files changed、diff stats

置信度：0.50-0.70 上限（数据有限）

使用场景：仅有 commit 数据无 PR 的缺陷，强制 needs_review=true

Workflow 执行模式

pipeline(pr_batches, classify, verify)
  ├─ classify: 15 defects/batch，Agent 读 batch JSON → 输出分类
  └─ verify:   对 confidence < 0.7 的条目做 adversarial verify
                (spawn 独立 Agent 尝试驳斥分类)

agent() for commit_batch — 单次调用，强制 needs_review: true

Output → agent_results.json → apply_agent_results.py → defect_db.json

🛡️ 质量审计体系

抽样策略

优先级策略数量
P0所有 needs_review 条目不计入限额
P1每类 audit_count 最低的 N 条（低 confidence 优先）5 × N
P2每类 audit_count 最低的 N 条（高 confidence 优先）5 × N
P3compatibility 双倍采样（已知 catch-all 类别）+N

日常: --sample-size 3 → ~30 条（5×3×2 + 3 + needs_review）
全量: --all --split 15（首次分类后 + 每月建议）

Audit Agent 审计范围

仅审计 5 个 Agent 产出字段（category、severity、root_cause、rationale、confidence），脚本字段不审。

还检查 is_npu_related：该缺陷是否真的与 Ascend NPU/CANN/HCCL 相关。

主 Agent 分组确认策略

组别变更内容确认方式原因
A 组category 变更逐条确认影响最大，category 决定 severity 等级和下游分析
B 组root_cause / rationale 重写批量采纳文字优化，审核价值低于 category
C 组severity / confidence 微调批量采纳数值调整，通常合理
D 组is_not_npu=true主 Agent 判断升级为 delete 还是保留 needs_review

争议解决机制（Step 3.5）

对 A 组中存疑的 category 变更：

  1. Spawn 1 个独立 Audit agent（不告知争议内容，只给 PR body + category 定义）
  2. 比较：原始分类 vs 第一次审计建议 vs 独立审计意见
  3. 主 Agent 做最终判决（基于论据质量，不做简单投票）
⚠ 关键规则： bug_id 必须原样抄写不得改动（曾出现 2025→2026 抄错）· action=accept 时 changes 为空，只写 evidence · action=change 时 changes 包含所有修正字段 · action=delete 需 ALL 4 条件满足才确认删除

💡 关键设计决策

1. PR-first 而非 commit-first

从 commit 消息提取信息量严重不足。PR 提供 body、reviews、labels、files 等多维度数据， 使 Agent 能做出高质量分类（confidence 0.8-0.95 vs 0.5-0.7）。100% PR body 覆盖。

2. Word-boundary 正则防止误匹配

"npu" 是 "input" 的子串，"cann" 是 "cannot" 的子串。\b 词边界防止这类误匹配。 这是从实际误报中总结出的关键修复。

3. 四维 NPU 校验而非单一维度

仅靠 label 校验会漏掉无标签 PR；仅靠 body 匹配容易匹配到样板文本。 四维联合校验（label + title tag + file path + body）平衡召回与精确。

4. 分层审计而非随机抽样

按 audit_count ASC 排序确保每个缺陷最终都被审计到（审计最少的优先）。 compatibility 双倍采样因为它已知是 catch-all 类别。低 confidence 优先因为更需要验证。

5. 分组确认而非逐条确认

A 组（category 变更）逐条确认因为影响最大；B/C 组批量采纳节省人力。 这平衡了质量保障与效率。

6. adversarial verify + 争议解决

对 confidence < 0.7 的条目 spawn 独立 Agent 尝试驳斥。对 category 变更存疑时 spawn 独立盲审 Agent，主 Agent 基于论据质量判决而非简单投票。

7. PR body 样板剥离

PR 的 "Accuracy Tests" 和 "Speed Tests" 章节中的 --device npu 启动命令是样板， 不代表实质性 NPU 讨论。在 body 检查前剥离这些章节。

8. 双路径 Agent 分类

PR 数据走深度分析（AGENT_PR_PROMPT，高置信度），commit-only 走浅层分析 （AGENT_COMMIT_PROMPT，低置信度 + 强制 needs_review）。避免对贫数据过度自信。
