1. 问题场景

1.1 触发条件

当用户通过 OpenAI 兼容 API 向多模态模型发送请求时，设置 n > 1（并行采样参数）会导致服务端报错：

curl http://localhost:30000/v1/chat/completions \
  -d '{
    "model": "Qwen2.5-VL-3B-Instruct",
    "messages": [{"role": "user", "content": [
      {"type": "image_url", "image_url": {"url": "https://example.com/photo.jpg"}},
      {"type": "text", "text": "描述这张图片"}
    ]}],
    "n": 3
  }'

1.2 参数 n 的含义

n 是 OpenAI Chat Completions API 的并行采样数量：对同一个 prompt 生成 n 个独立 completion，每次采样从 logits 分布随机抽取 token，因此 n 条路径自然分叉产生不同输出。

Prompt
+ 🖼️ 图片
→
Choice[0]
"熊猫正在吃竹子…"
→
Choice[1]
"图中一只熊猫坐着…"
→
Choice[2]
"照片展示熊猫在竹林…"

关键点：n 个采样共享同一个 prompt，包括同一份图片数据。服务端内部会浅拷贝请求对象 n 次，每个 replica 引用同一个图片 tensor → 共享同一块共享内存。

1.3 现象对比

修复前 (n=3)修复后 (n=3)
结果 ❌ 报错 ✅ 3 个 choices 正确返回
根因 共享内存被第一个 replica 提前 unlink 引用计数保护，所有 replica 完成后才释放

2. 根因分析

2.1 完整数据流

┌── TokenizerManager 进程 ──┐         ┌── Scheduler 进程 ────────┐
│                            │         │                            │
│ 1. _tokenize_one_request  │         │                            │
│    └─ 图片 tensor (CPU)   │         │                            │
│                            │         │                            │
│ 2. n=3 → 浅拷贝 ×3         │         │                            │
│    replica-1 ┐             │         │                            │
│    replica-2 ├─ 同一 tensor │         │                            │

2.2 根本原因

文件 python/sglang/srt/managers/mm_utils.py，类 ShmPointerMMData 的 materialize() 方法：

# 修复前 — 无条件 unlink
def materialize(self) -> torch.Tensor:
    tensor = self.tensor.clone()
    if self._shm_handle is not None:
        self._shm_handle.close()
        try:
            self._shm_handle.unlink()    # ← 第一个 replica 就删除了共享内存
        except FileNotFoundError:
            pass                         # ← 后续 replica 访问已被删除的 shm → 崩溃
        self._shm_handle = None
    return tensor

核心问题：同一个 shm_name 被 n 个 replica 共享，但 unlink() 在第一次 materialize() 时就被调用，导致后续 replica 无法访问共享内存中的数据。POSIX shm_unlink() 虽然只删除名字（已有 fd 仍可读写），但在某些环境下（特定内核配置、Windows、Ascend 驱动实现等），时序竞争会导致数据不可用。

2.3 风险模式抽象

🔴 通用模式：一份共享资源 + 多个消费者 + 无引用计数 → 资源提前释放

这个模式不仅影响 n > 1，理论上也影响：

  • TP > 1：每个 rank 各打开一次共享内存，rank 0 的 materialize() 可能在 rank 1 还没 shm_open 时就 unlink
  • n > 1 × TP > 1：消费者数 = n × TP，竞争更激烈
  • Batch + n > 1：_handle_batch_request 的 n>1 分支（L1459）发送多个独立的 _send_one_request

3. 修复方案分析

3.1 修改内容

位置修改说明
ShmPointerMMData 类体 新增 _shm_ref_counts = {} 类级别字典，按 shm_name → 引用计数
__init__ 新增 _shm_ref_counts[self.shm_name] = 1 创建者持有一个初始引用
__getstate__ 新增 _shm_ref_counts[self.shm_name] += 1 每次 pickle 序列化（发送给一个 replica）时递增
materialize 递减计数 → ≤0 时才 unlink 仅最后一个消费者释放共享内存

3.2 修复后的 materialize()

# 修复后 — 引用计数保护
def materialize(self) -> torch.Tensor:
    tensor = self.tensor.clone()
    if self._shm_handle is not None:
        self._shm_handle.close()
        # 递减引用计数，仅最后一个消费者 unlink
        if self.shm_name in ShmPointerMMData._shm_ref_counts:
            ShmPointerMMData._shm_ref_counts[self.shm_name] -= 1
            if ShmPointerMMData._shm_ref_counts[self.shm_name] <= 0:
                try:
                    shared_memory.SharedMemory(name=self.shm_name).unlink()
                    del ShmPointerMMData._shm_ref_counts[self.shm_name]
                except FileNotFoundError:
                    pass
        self._shm_handle = None
    return tensor

3.3 引用计数修复的深度分析

修复的核心思路是正确的，但在引用计数的生命周期管理上存在几个需要关注的边界情况。

3.3.1 引用计数生命周期追踪

以最简单的 n=3 + TP=1 场景为例，追踪 _shm_ref_counts 的完整变化：

# TokenizerManager 进程 (发送端)
ShmPointerMMData.__init__()          → ref[shm_A] = 1   # 创建
__getstate__()  (replica-1 pickle)   → ref[shm_A] = 2   # ZMQ 序列化
__getstate__()  (replica-2 pickle)   → ref[shm_A] = 3
__getstate__()  (replica-3 pickle)   → ref[shm_A] = 4

# Scheduler 进程 (接收端) — 注意：不同进程！
# 此进程有自己独立的 _shm_ref_counts，shm_A 不在其中！
replica-1: __setstate__() → shm_open(shm_A)
replica-1: materialize()  → ref 中无 shm_A → 跳过递减 → 跳过 unlink
replica-2: materialize()  → 同上
replica-3: materialize()  → 同上

# 结果：发送端 ref=4 永不归零，接收端 ref 中无记录
→ 共享内存 shm_A 永远不被 unlink → 内存泄漏！
🔴 P0 — 引用计数仅在同一进程内有效

_shm_ref_counts 是类级别的 Python dict，仅存在于单个 Python 解释器进程中。__getstate__ 在发送端（TokenizerManager）递增，但 materialize() 在接收端（Scheduler）递减——两端是不同的进程，各自有独立的 ref dict。接收端的 ref dict 中没有 shm_A 的条目，递减逻辑被跳过，shm 永远不会被 unlink。

为什么 Bug 仍然被修好了？ 因为之前 n=1 时 materialize() 无条件 unlink（工作正常）；n>1 时第一个 replica unlink 后别的 replica 访问不到数据（💥）。现在改为引用计数后，n=3 时 没有一个 replica 会 unlink（ref 不在 dict 中），数据始终可用——但也始终泄漏。这是一个"用内存泄漏换正确性"的取舍。

3.3.2 有 TP broadcast 时的引用计数

当 TP > 1 时，流程中多了一次 broadcast_pyobj，引用计数在同一进程内多了一次 __getstate__：

# Scheduler rank 0 进程
replica: __setstate__() → shm_open(shm_A)
# broadcast_pyobj 到 rank 1, 2, 3 ...
broadcast_pyobj → __getstate__()  # rank 0 上再次 pickle → 如果 _shm_ref_counts 中有 shm_A 则 ++
# 但 rank 0 的 ref dict 中同样没有 shm_A（由 ZMQ 接收时 __setstate__ 不创建条目）

TP broadcast 同样不会在接收端的 ref dict 中创建条目，引用计数链路依旧断裂。

3.3.3 完整的风险清单

🔴 P0 — 跨进程引用计数断裂（上述 3.3.1）

发送端递增、接收端递减，两端是不同进程 → 共享内存泄漏是必然的，只是被原 Bug 的错误行为掩盖了。正确做法应是在 __setstate__ 中初始化 ref 计数，或在接收端也维护计数。

🔴 P0 — 单次发送场景 ref 永远不归零

即使不考虑跨进程问题，审视同一进程内的计数逻辑：__init__ → ref=1，__getstate__ → ref=2，materialize() → ref=1。之后 ref 永远停留在 1，共享内存同样不会释放。n=1 的正常路径也存在这个"静默泄漏"。

🟡 P1 — __del__ 未递减引用计数

当前 __del__ 只 close handle 不 unlink。如果某个 replica 对象被垃圾回收但从未调用 materialize()（例如请求被 abort、异常路径），对应的引用计数不会被递减，导致其他正常 consumer 永远等不到 ref ≤ 0 的时刻。

🔴 P0 — 原始对象的引用泄漏

在 TokenizerManager 发送端，原始 ShmPointerMMData 对象在 _send_one_request 返回后不再被引用，由 GC 回收。__del__ 只 close 不 unlink，且不递减引用计数。原始对象持有的 ref=1 永远无法释放。

🟡 P1 — 线程安全

_shm_ref_counts 是普通 dict，无锁保护。多线程并发 __getstate__ 或 materialize() 可能产生竞态条件。

3.3.4 建议的修复方向

# 方案 A：在 __setstate__ 中初始化接收端引用计数
def __setstate__(self, state):
    ...
    self._shm_handle = shared_memory.SharedMemory(name=self.shm_name)
    # 接收端首次打开 shm 时创建计数条目
    ShmPointerMMData._shm_ref_counts.setdefault(self.shm_name, 0)
    ...

# 方案 B：放弃引用计数，用更简单的"n 副本感知"方案
# 发送端预先知道 n 的值，在 ShmPointerMMData 初始化时告知 expected_consumers
# consumer_id 通过 ZMQ message 传递，materialize 时检查是否最后一个 consumer

# 方案 C：在 __del__ 中补齐计数递减 + 条件 unlink
def __del__(self):
    if getattr(self, "_shm_handle", None) is not None:
        self._shm_handle.close()
        self._shm_handle = None
    # 安全网：如果 materialize 从未被调用，del 时递减计数
    if hasattr(self, "shm_name") and self.shm_name in ShmPointerMMData._shm_ref_counts:
        ShmPointerMMData._shm_ref_counts[self.shm_name] -= 1
        if ShmPointerMMData._shm_ref_counts[self.shm_name] <= 0:
            try: shared_memory.SharedMemory(name=self.shm_name).unlink()
            except: pass

4. NPU 平台影响分析

4.1 NPU 的传输模式

关键函数 _determine_tensor_transport_mode()：

def _determine_tensor_transport_mode(server_args: ServerArgs) -> TensorTransportMode:
    is_cross_node = server_args.dist_init_addr
    if is_cross_node:
        return "default"       # 跨节点 → 内联传输，不经过共享内存
    else:
        return "cuda_ipc"      # 单节点 → 共享内存传输

在 wrap_shm_features() 中：

def wrap_shm_features(obj):
    if _get_is_default_transport() or ...:
        return obj              # "default" 模式 → NO-OP，直接返回
    # "cuda_ipc" 模式 → ShmPointerMMData 包装激活
    for item in obj.mm_inputs.mm_items:
        item.feature = _wrap_tensor_or_list(item.feature)  ← 这里创建 ShmPointerMMData
结论：NPU 单节点 CI（所有现有 nightly-*-npu-a3 runner）的 dist_init_addr 为 None，→ 传输模式为 "cuda_ipc" → ShmPointerMMData 路径完全激活 → PR #23685 修复的代码路径在 NPU 上切实执行。

4.2 NPU 独有代码路径

在 mm_utils.py:773-774 中有一个 NPU 专属同步点：

def get_embedding_and_mask(...):
    ...
    embedding, input_ids = _get_chunked_prefill_embedding(...)
    if _is_npu:
        torch.npu.current_stream().synchronize()  # ← NPU ONLY
    special_multimodal_mask = _get_multimodal_mask(input_ids, placeholder_tensor)

这行 synchronize() 确保 NPU stream 上的多模态 embedding 计算完成后，才开始生成 mask。如果此同步缺失或时序不当，会导致 embedding 和 mask 位置对不上。这个路径 仅在 NPU 上执行，CUDA CI 无法覆盖。

class TestShmPointerMMDataFix(CustomTestCase):
    """验证 PR #23685 引用计数修复的各种边界条件"""

    def test_multi_pickle_same_process(self):
        """同进程 n=3 模拟：pickle 3 次 → 3 次 materialize → 验证最后一次释放 shm"""
        tensor = torch.randn(100, 100)

        # 创建 → ref=1
        shm = ShmPointerMMData(tensor)
        name = shm.shm_name

        # pickle 3 次（模拟 n=3 发送）→ ref=4
        states = [shm.__getstate__() for _ in range(3)]

        # 3 次 materialize → ref=1, 0?? unlink?
        for i, state in enumerate(states):
            replica = ShmPointerMMData.__new__(ShmPointerMMData)
            replica.__setstate__(state)
            result = replica.materialize()

8. 总结与路线图

8.1 关键发现

  1. PR #23685 修复了一个真实 Bug 但遗留了设计缺陷：引用计数方案在同进程内有效（解决了 n 个 replica 并发 materialize 的竞争），但在跨进程场景下（TokenizerManager → Scheduler 的 ZMQ 传输，以及 TP broadcast）引用计数链路断裂，导致共享内存静默泄漏。详细分析见 3.3 节。
  2. NPU VLM 测试只有一种形态：100% 的 NPU VLM 测试是 MMMU 精度跑分（n=1, temp=0, batch eval）。这不是功能测试——它验证"精度不退化"，验证不了"功能不崩溃"。
  3. NPU 存在独有代码路径但无专属保护：_is_npu 同步点是 NPU-only，正确性仅靠 MMMU 精度间接验证。
  4. 13 个 VLM 功能维度在 NPU 上完全空白：单图/多图/视频/音频/流式/混合 batch/precomputed/processor_output/penalty/function calling/matched stop/HiCache/LoRA —— 全部未测。
  5. 测试成本可控：Phase 1 + 1.5 增量为 ~315 秒（约 5 分钟），不增加新的 CI runner，全部复用现有 nightly-4-npu-a3。其中 Phase 1.5 的 ShmPointerMMData 边界条件测试可作为技术债可视化标记——预期部分 FAIL，直接暴露引用计数的残余问题。

8.2 一句话总结

NPU 多模态测试当前是"精度看门狗"而非"功能防护网"。 PR #23685 暴露的 Bug 正好落在两者的真空地带—— 精度测试（n=1, temp=0）永远无法触发它， 而功能测试（n=3, stream=True）根本不存在。
