本文档解决风险因子中最关键的数据源——历史缺陷——如何采集、分类、存储和持续更新。它是整个风险系统的"感知层"，其他四份文档依赖它的输出。

1核心设计决策：为什么不是"全量分析每次重挖"

❌ 每次全量分析重挖

  • ▸ 每次跑 git log --grep 扫描 25000+ commits
  • ▸ 耗时线性增长: 今天 25000 → 6月后 30000 → 1年后 36000
  • ▸ 99% 的 commit 上次已经分析过，重复计算浪费
  • ▸ Agent 重复调用做同样的分类判断
  • ▸ 全量分析变重 → 不愿意频繁跑 → 数据变旧
  • ▸ 缺陷数据和代码分析耦合 → 改一个文件要重新挖全量历史

✅ 两阶段架构

  • ▸ Phase A: 历史深挖只跑一次（M0 初始化）
  • ▸ Phase B: 增量只扫描 {last_scan}..HEAD（通常 0-20 个新 commit）
  • ▸ 耗时恒定: 每天增量扫描 <2s
  • ▸ Agent 只分析新 commit，不重复
  • ▸ 缺陷 DB 独立维护 → 随时可以查询，不受全量分析频率约束
  • ▸ 全量分析和缺陷采集完全解耦

🧠 关键洞察

缺陷数据是单向累积的不可变历史。已存在的 bug fix commit 永远不会消失，永远不会改变。所以"重新挖掘历史"是纯粹的浪费——只有新出现的 commit 值得分析。这与代码风险评分完全不同（代码随时在变，需要重算）。

2两阶段架构全景

✅ 关键设计原则

单一写入者: 只有增量追加进程能写 defect_db.json。全量分析、增量评估、KG 构建都是只读消费者。这消除了并发写入冲突——你不需要锁、不需要事务、不需要担心两个分析流程同时修改同一条缺陷记录。

3数据源选择：PR 优先，Commit 兜底

3.1 为什么这是架构级决策

缺陷挖掘的数据源选择直接影响 Agent 分类的置信度、信息量和维护成本。选错了不是"效果差一点"，而是"Agent 从阅读理解变成猜谜"——confidence 从 0.9 跌到 0.5，大量记录需要人工复核。

📝 Commit-Only 方式的局限

  • ▸ 只有 1 行 title + 可选 body（通常为空）
  • ▸ 没有复现步骤 → Agent 无法判断严重度
  • ▸ 没有讨论 → 无法知道影响范围
  • ▸ 没有标签 → 无法快速分类
  • ▸ 没有 review → 丢失"差点成 bug"的信号
  • ▸ Agent 从 3 行文本推断全部信息
  • ▸ 典型 confidence: 0.5-0.7

✅ PR-First 方式的优势

  • ▸ Description: 问题/复现/根因 三段式
  • ▸ Review comments: reviewer 的质疑暴露隐藏风险
  • ▸ Labels: bug/critical/NPU 自动分类
  • ▸ Linked issues: 用户报告的问题上下文
  • ▸ Timeline: labeled/milestoned/merged 时间线
  • ▸ Near-miss 信号: review 中拦截的潜在 bug
  • ▸ Agent 做"阅读理解"非"考古猜测"
  • ▸ 典型 confidence: 0.8-0.95

3.3 但 Commit 不能被完全替代

两种场景下 commit 是必要的兜底：

场景 1: 老仓库历史

sglang 早期可能没有强制 PR 流程，或 PR 已因 branch 清理而丢失。Commit 是 Git 中唯一永久保留的记录——永远不会丢失。

场景 2: PR 信息本身不足

有些 PR 的 description 就是一句 "fix bug"，review 也只有 "LGTM"。这种情况下 PR 并不比 commit 更有价值——走 commit 浅层分析即可。

3.6 PR Review 的 Near-Miss 信号——Commit 永远做不到的事

PR review comments 中有一种独特的信息：被 reviewer 拦截的潜在 bug。这不是已经发生的缺陷，而是"如果 reviewer 没注意到，它就是下一个 BUG-2026-XXX"。

🔮 Near-Miss 示例

PR #24592 — Review Comment Thread:
Reviewer A: "This approach looks correct, but did you consider
  that CANN 8.5.0 changed the default stream behavior?
  Under heavy load, this could cause a race condition in
  the dual-stream dispatch path."

Author: "You're right — I didn't catch that. I'll add a stream
  sync at the dispatch boundary. Good catch."

Author: [commits fix] "Added torch.npu.synchronize() before
  token dispatch in dual-stream path."

这个 thread 告诉你：① fused_moe_method_npu.py 在多 stream 场景下容易出错 ② CANN 版本升级是触发因素 ③ 如果没有 review，这就是一个 silent_corruption 类型的线上 bug。Agent 可以把这类信号提取为低权重的风险标记。

Near-Miss → Risk Signal 的结构化输出

{
  "type": "near_miss",
  "source_pr": 24592,
  "source_comment_url": "https://github.com/sgl-project/sglang/pull/24592#discussion_r...",
  "risk_pattern": "CANN version API change + stream synchronization gap",
  "module": "fused_moe_method_npu.py",
  "symptom_if_not_caught": "silent_corruption",
  "would_affect_features": ["ascend_fuseep", "dual_stream_moe"],
  "trigger_condition": "CANN 8.5.0 + dual-stream enabled + heavy load",
  "caught_by": "code_review",
  "reviewer_comment_sentiment": "concerned",  // reviewer 表达了担忧
  "weight": 0.3,  // 不是实际缺陷，但提示该区域需额外测试关注
  "recommendation": "Add stress test for dual-stream dispatch under simulated heavy load"
}

Near-Miss 的 weight:0.3 意味着它不会直接推高风险评分，但会在 KG 图中形成一条虚线关系边，当多个 near-miss 指向同一模块时，累积效应会触发告警。这是纯 commit 分析永远无法获得的信息维度。

3.7 实现脚本

#!/bin/bash
# .sglang-risk/defect_mining/enrich_with_pr.sh
# 对每个候选 commit 查询关联 PR

while IFS='|' read -r sha date msg; do
  # gh api 查询包含此 commit 的 PR
  PR_DATA=$(gh api "search/issues?q=repo:sgl-project/sglang+type:pr+sha:${sha}" \
    --jq '.items[0] | {number, title, body, labels: [.labels[].name]}' 2>/dev/null)

  if [ -n "$PR_DATA" ] && [ "$PR_DATA" != "null" ]; then
    PR_NUM=$(echo "$PR_DATA" | jq -r '.number')
    PR_BODY=$(echo "$PR_DATA" | jq -r '.body')
    LABEL_COUNT=$(echo "$PR_DATA" | jq -r '.labels | length')

    # 判断 PR 质量: description 长度 + labels 数量
    BODY_LEN=$(echo "$PR_BODY" | wc -c)
    if [ "$BODY_LEN" -gt 200 ] || [ "$LABEL_COUNT" -gt 1 ]; then
      # 高质量 PR → 获取 review comments
      REVIEWS=$(gh api "repos/sgl-project/sglang/pulls/${PR_NUM}/comments" \
        --jq '[.[] | {user:.user.login, body, path, diff_hunk}]' 2>/dev/null)

      TIMELINE=$(gh api "repos/sgl-project/sglang/issues/${PR_NUM}/timeline" \
        --jq '[.[] | select(.event=="labeled") | .label.name]' 2>/dev/null)

      echo "{\"source\":\"pr\",\"quality\":\"high\",\"pr_number\":$PR_NUM, ...}" >> enriched.json
    else
      # 低质量 PR → 降级为 commit 分析
      echo "{\"source\":\"commit\",\"quality\":\"low\",\"sha\":\"$sha\", ...}" >> enriched.json
    fi
  else
    # 无 PR → commit only
    echo "{\"source\":\"commit\",\"quality\":\"fallback\",\"sha\":\"$sha\", ...}" >> enriched.json
  fi
done < candidates.txt

⚠️ GitHub API 速率限制

Phase A 历史深挖时可能一次查询数百个 commit。GitHub API 限制为 5000 req/h（认证用户）。建议对批量查询使用 gh search issues 的批量模式，或缓存已查询的 PR 结果。Phase B 增量追加每天只查 0-5 个新 commit，不受限。

Step 1: 双重过滤的 grep 策略

不是简单的单次 grep。用两步策略最大化召回率同时控制噪音：

# 第一遍: 宽松召回（最大化覆盖）
# 匹配 "fix/bug/repair/correct/patch + 出现在 NPU 相关文件中"
git log --all --since="2024-01-01" \
  --grep="fix\|bug\|repair\|correct\|patch\|resolve\|address" \
  -- "python/sglang/srt/hardware_backend/npu/**" \
     "python/sglang/srt/layers/moe/token_dispatcher/fuseep.py" \
     "python/sglang/srt/layers/quantization/**" \
     "python/sglang/srt/disaggregation/ascend/**" \
     "python/sglang/srt/lora/backend/ascend_backend.py" \
  --format="%H|%ai|%s" \
  > raw_primary.txt

# 第二遍: 补充跨模块修复（commit message 提到 NPU 但文件不在 npu/ 下）
git log --all --since="2024-01-01" \
  --grep="npu\|ascend\|NPU\|Ascend\|CANN\|HCCL\|torch_npu\|sgl_kernel_npu" \
  --format="%H|%ai|%s|%b" \
  > raw_secondary.txt

# 合并去重
cat raw_primary.txt raw_secondary.txt | sort -u > raw_all.txt

Step 2: 噪音过滤规则

很多匹配到的 commit 不是真正的缺陷修复——比如添加新功能的 commit 也包含 "fix typo"、重构代码等。规则过滤：

规则示例动作
commit message 含 "typo" / "format" / "lint" / "doc" / "comment""fix typo in readme"剔除
只改了 .md / .rst / .txt 文件-- "docs/**" only剔除
commit message 含 "cherry-pick" / "revert"重复或回退标记去重
改动行数 < 5（仅 import 调整等）trivial change低优先级
含 "bump version" / "upgrade dep"版本升级，非 bug fix剔除

Step 3: Agent 并行分类

过滤后通常剩余 50-100 条候选记录。启动 3-5 个并行 Agent，每个分析 15-30 条：

// Workflow 片段
phase('Defect Mining: Agent Classification')
const rawEntries = parseRawDefects('raw_filtered.txt')

const classified = await parallel(
  chunk(rawEntries, 3).map((batch, i) => () =>
    agent(DEFECT_CLASSIFY_PROMPT(batch), {
      label: `defect:batch${i}`,
      schema: DEFECT_SCHEMA  // 强制结构化输出
    })
  )
)

// 合并 + 去重（同一 bug 可能被多个 commit 修复）
const deduped = deduplicateByFilesAndFeature(
  classified.filter(Boolean).flatMap(r => r.defects)
)

4.2 Agent 分类 Prompt 设计

DEFECT_CLASSIFY_PROMPT:
  "你是一个 NPU 缺陷分类专家。分析以下 git commit 记录，
   判断它们是否代表一个真实的 NPU 相关缺陷修复。

   对每条记录，你需要:
   1. 判断是否为真实缺陷修复（vs 新功能/重构/文档/格式）

           silent_corruption | compatibility | compile_error

   输出 JSON Schema: { defects: [{ bug_id, category, severity, ... }] }"

6Phase B: 增量追加（Daily Cron · 持续）

5.1 极简流程

#!/bin/bash
# .sglang-risk/scripts/defect_append.sh
# 每天 04:00 UTC 执行

DB=".sglang-risk/db/defect_db.json"
LAST=$(jq -r '._meta.last_scan_commit' "$DB")

# 只扫新 commit
NEW_COMMITS=$(git log ${LAST}..HEAD \
  --grep="fix\|bug\|npu\|ascend\|CANN" \
  --format="%H|%ai|%s" \
  -- "python/sglang/srt/")

if [ -z "$NEW_COMMITS" ]; then
  echo "No new commits to scan."
  exit 0
fi

# Agent 增量分类
python -c "
from defect_miner import classify_incremental, append_to_db
new_defects = classify_incremental('''$NEW_COMMITS''')
append_to_db('$DB', new_defects)
"

⚠️ 边界情况处理

force rebuild 场景: 如果怀疑 defect_db.json 数据有问题（如 Agent 分类错误），提供 --force-rescan 标志，清空 DB 重新运行 Phase A。这个场景极少发生（几个月一次），不需要为此把每次全量分析都绑上全量缺陷重挖。

7消费者集成：三个下游如何消费 defect_db

📊 消费者 1: 全量分析

用途: 填充 risk_profiles 的 historical_defects 因子

# 全量分析读取缺陷数据
def get_historical_defect_score(feature_name):
    defects = load_defect_db()
    relevant = [
        d for d in defects
        if feature_name in d['features_affected']
    ]
    # 评分: 最近90天缺陷数 × 平均严重度
    recent = [d for d in relevant
              if days_since(d['date_fixed']) <= 90]
    if not recent: return 1
    avg_severity = mean(d['severity'] for d in recent)
    return min(5, len(recent) * avg_severity / 3)

🔄 消费者 2: 增量评估

用途: PR check 时查询受影响特性是否有历史缺陷

# PR 修改了 fuseep.py
# 查询: 这个文件历史上出过什么 bug？
def check_defect_history(changed_files):
    defects = load_defect_db()
    warnings = []
    for f in changed_files:
        file_defects = [
            d for d in defects
            if any(f.endswith(ff) for ff in d['files_fixed'])
        ]
        if file_defects:
            # 按 failure_mode 聚类
            modes = Counter(
                d['failure_mode'] for d in file_defects
            )
            warnings.append({
                'file': f,
                'past_defects': len(file_defects),
                'common_modes': modes.most_common(3)
            })
    return warnings

🧩 消费者 3: KG 图构建

用途: 创建 Defect 节点 + AFFECTED_BY 和 FIXED_IN 边

# build_graph.py 中读取缺陷
def add_defect_nodes(G, defect_db_path):
    defect_db = json.loads(Path(defect_db_path).read_text())
    for d in defect_db['defects']:
        G.add_node(d['bug_id'],
            type='Defect',
            category=d['category'],
            severity=d['severity'],
            is_silent=d['is_silent'],
            failure_mode=d['failure_mode']
        )
        # AFFECTED_BY: Defect → Feature
        for feat in d['features_affected']:
            if G.has_node(feat):
                G.add_edge(d['bug_id'], feat,
                    type='AFFECTED_BY',
                    weight=d['severity']
                )
        # FIXED_IN: Defect → SourceFile
        for file_path in d['files_fixed']:
            file_id = file_path.split('/')[-1]
            if G.has_node(file_id):
                G.add_edge(d['bug_id'], file_id,
                    type='FIXED_IN', weight=0.5
                )

8Agent 分类 Prompt 详细设计

这是整个链路中最依赖 Agent 质量的环节。Prompt 需要包含分类标准 + NPU 领域知识 + 输出约束。

你是一个 NPU（华为 Ascend）平台的缺陷分析专家。

## 任务
分析以下 git commit 记录，判断哪些是真实的 NPU 缺陷修复。
对每条确认为缺陷的记录，进行结构化分类。

## 分类标准

### Category（缺陷类型）
- precision_loss: 数值精度丢失（如 FP32→BF16 误差，quant scale 漂移）
- crash: 程序崩溃（segfault, CUDA illegal memory, HCCL timeout）
- perf_regression: 性能退化（throughput 下降 >10%）
- memory_leak: 内存泄漏或 OOM
- silent_corruption: 静默数据损坏（无 crash，输出悄悄变错）
- compatibility: 版本兼容问题（CANN/torch_npu 升级导致）
- compile_error: 编译/构建问题

### Severity（严重度 1-5）
5 = 静默错误（用户无感知但结果错误，最危险）
4 = crash 或 OOM（用户立即感知但影响面大）
3 = 精度退化（可检测到但可能被忽略）
2 = 性能退化
1 = 编译/构建问题

### Root Cause（根因分类）
- format_cast: FRACTAL_NZ 格式转换问题
- quant_scale: 量化 scale 计算错误
- stream_sync: 多 stream 同步问题
- graph_capture: NPU Graph 捕获/重放问题
- hccl_timeout: HCCL 通信超时
- cann_api: CANN API 使用错误
- memory_layout: 内存布局/stride 问题
- token_routing: MoE token dispatch 路由错误

### is_silent 判断标准
如果满足以下条件之一，标记为静默错误:
- commit message 提到 "wrong output" / "incorrect result" / "精度"
- 但没有任何 crash/error/timeout 关键词
- 且修复不涉及异常处理逻辑

## 输入数据
{raw_commit_entries}

## 输出格式
严格按照以下 JSON Schema:
{
  "defects": [
    {
      "bug_id": "BUG-YYYY-NNN",
      "commit_sha": "...",
      "category": "silent_corruption",
      "severity": 5,
      "is_silent": true,
      "failure_mode": "wrong_output",
      "root_cause": "quant_scale",
      "features_affected": ["ascend_fuseep", "dynamic_quant"],
      "test_would_catch": true,
      "test_type_needed": "oracle",

}

💡 持续优化的 Prompt

confidence 低的记录是 Prompt 优化的主要信号。定期（每月）review confidence < 0.7 的记录，找出 Agent 反复分错的模式，补充到 Prompt 的"分类标准"中。本质上这是一个human-in-the-loop 的持续学习循环。

10与全量/增量管线的时序关系

✅ 时序解耦

Phase A 只在 M0 初始化时跑一次。之后缺陷采集（daily）和全量分析（weekly）完全独立运行，互不阻塞。全量分析读 DB 时可能恰好增量进程在写——但因为是单向追加（append-only），不会发生数据竞争。

11总结

⛏️ Phase A

  • 全量历史深挖
  • M0 初始化跑一次
  • 双重 grep 过滤
  • 3-5 Agent 并行分类
  • 人工抽检低置信度
  • 产出 seed defect_db

📥 Phase B

  • 增量追加
  • Daily cron 自动运行
  • 仅扫描 {last}..HEAD
  • 1 Agent 逐条分析
  • <5s 完成
  • defect_db 单向追加

🗄️ 共享中间层

  • 单一写入者 (Phase B)
  • 多消费者只读
  • 全量分析读 DB 算评分
  • 增量评估读 DB 做告警
  • KG 读 DB 建节点+边
  • 完全解耦 · 零写冲突

🎯 一句话结论

历史深挖跟 M0 全量分析一起跑（首次建库），之后增量追加作为 daily cron 独立运行。defect_db.json 是所有分析流程的只读数据源，唯一的写入者是增量追加进程。
