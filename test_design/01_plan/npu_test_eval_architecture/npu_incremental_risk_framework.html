NPU 风险测试框架 — 增量评估架构

从"一次性快照"到"持续演进的风险画像"——每次代码/测试变更自动触发增量更新

📐 设计目标: 可增量 · 可对比 · 可追溯 🔄 触发机制: git hook + CI scheduled + PR check 📊 输出: Risk Delta Report (变化量，非全量) 💾 持久化: Baseline JSON → Delta JSON → Trend DB

1问题定义：为什么一次性分析不够

全量分析的痛点

  • 每次运行都从头扫描 — 100+ 文件全部重读，即使 99% 未变
  • 无法回答"趋势"问题 — 风险是上升还是下降？测试覆盖率是改善还是恶化？
  • 结果之间无法对比 — 两次分析之间没有 diff，看不出变化
  • CI 集成困难 — 全量分析 3-5min 太长，不适合 PR 门禁
  • 历史缺陷数据的时效性 — git log 每次重新 grep，没有增量追加

增量分析的期望

  • ✅ 只分析变更部分 — 1 个文件改了，只重算那 1 个文件的风险
  • ✅ 输出 Delta — "FuseEP 风险评分 +3↑ 因为新增 dual-stream 分支"
  • ✅ 趋势追踪 — 本月 vs 上月，测试覆盖率趋势线
  • ✅ PR 门禁可集成 — 增量分析 < 30s，可作为 PR comment
  • ✅ 缺陷数据增量追加 — 不重复 grep 全量 git log

2持久化数据模型设计

增量评估的前提是有东西可以对比。每次分析后，结构化结果持久化到仓库中，作为下一次分析的 baseline。

🔑 Fingerprint 机制

每个特性生成一个内容指纹（SHA256 of 归一化后的源代码 + 测试覆盖数据）。下次分析时，对比指纹即可 O(1) 判断是否需要重算。指纹不变 → 跳过该特性的全部 Agent 分析，直接复用上次结果。

依赖图维护策略

  • 全量分析时自动生成 — Phase 1 Agent 输出中包含文件的 feature 归属
  • 增量时优先复用 — 新文件不在 map 中 → 启动 1 个 Agent 专门分析新文件
  • map 本身也做版本管理 — dependency_map.json 随代码变更一起更新
  • 保守策略 — 不确定时标记为 "可能需要重算"，宁可多算不可漏算

3.3 PR 门禁模式（最快路径）

当集成到 CI 作为 PR check 时，走一条极简路径：

1

git diff 检测

获取 PR 变更文件列表，匹配 dependency_map → affected_features。如果 affected_features 为空（如只改了文档），直接返回 "No risk impact"。

2

Fingerprint 快速跳过

对 affected_features 做 fingerprint 比对。如果源码内容指纹与 baseline 一致（如只有注释/格式变更），跳过。

3

增量风险评分

仅对指纹变化的特性，启动 1 个 Risk-Scorer Agent 重新评分。这个 Agent 只读变更文件 + baseline 中其他文件的数据（不重读全量）。

4

输出 Delta Comment

生成精简版 delta：风险上升的特性 + 风险下降的特性 + 建议行动。作为 PR comment 自动发布。

< 30s
典型 PR 门禁耗时
1-5

4变更分类与对应的增量策略

不同类型的变更对风险评估的影响路径不同。框架需要识别变更类型，然后走精准的增量路径。

变更类型检测方式影响的风险因子增量策略Agent
源码逻辑变更 *.py 在 python/sglang/srt/ 下 code_complexity, platform_divergence 对该文件所属 feature 重新评分 Risk-Scorer
新增环境变量/配置 environ.py diff 中新增 SGLANG_NPU_* code_complexity, platform_divergence 判断是否属于已有 feature 还是新 feature Feature-Discovery
新增测试文件 test/registered/ascend/ 下新增 *.py test_effectiveness 分析测试类型和质量 → 更新 test_effectiveness 评分 Test-Evaluator
修改现有测试 test/registered/ascend/ 下修改 *.py test_effectiveness 仅重新评估该测试的质量评分 Test-Evaluator
Bug Fix commit commit message 含 fix/bug + NPU 关键词 historical_defects 追加缺陷记录 → 更新相关 feature 的 historical_defects 评分 Defect-Recorder
依赖版本升级 pyproject_npu.toml / requirements*.txt dependency_depth 分析依赖变更的影响范围 → 标记相关 feature Dependency-Analyzer
CI 配置变更 .github/workflows/*npu*.yml （间接）test_effectiveness 如果 CI 覆盖范围变了（增/减 suite），更新对应特性 CI-Analyzer
文档/注释变更 *.md / 注释行 diff 无 指纹比对命中 → 直接跳过 无

🎯 关键设计原则：变更粒度匹配

不是每种变更都需要启动 Agent。文档变更 → 直接跳过。测试新增 → 只更新一个风险因子。核心逻辑变更 → 重新评分。这个分级机制确保增量评估的计算量严格正比于变更的严重程度。

5CI 集成方案

5.1 三种运行模式

⚡ PR Check（门禁）

每次 PR 自动触发 · 增量模式 · <30s

  • git diff → affected_features
  • 指纹比对跳过未变特性
  • 仅重算变更特性（1-5个）
  • 输出: Delta Comment + Merge Gate
  • 阻断条件: 新增特性风险评分 ≥ critical 且无对应测试

🕐 Scheduled（定时全量）

每周/每日定时 · 全量模式 · 3-5min

  • 每周一次全量基线刷新
  • 重新计算所有 18 个特性的完整风险画像
  • 更新 baseline snapshot
  • 生成周度趋势报告: 风险↑/↓/→
  • 输出: Risk Dashboard 数据

🔧 Manual（手动深挖）

按需触发 · 深度模式 · 5-8min

  • 对指定特性做 L0 深度分析
  • Bug 考古 + 跨平台差分 + Fuzz 建议
  • 用于 release 前的专项审计
  • 新特性合入后的首次评估
  • 输出: 指定特性的完整测试策略

5.3 GitHub Actions Workflow 片段

# .github/workflows/npu-risk-check.yml
name: NPU Risk Delta Check

on:
  pull_request:
    paths:
      - 'python/sglang/srt/**'
      - 'test/registered/ascend/**'
      - 'python/sglang/srt/environ.py'

jobs:
  risk-delta:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # 需要 git history 做 diff

      - name: Compute Risk Delta
        run: |
          python .sglang-risk/scripts/run_delta.py \
            --baseline origin/main \
            --head HEAD \
            --mode pr_check

      - name: Post Delta Comment
        uses: actions/github-script@v7
        with:
          script: |
            const delta = require('./.sglang-risk/deltas/latest.json');
            const riskIncreased = delta.summary.risk_increased.length > 0;
            if (riskIncreased && delta.changes.some(c => c.new_level === 'critical')) {
              core.setFailed('Critical risk increase detected. See comment for details.');
            }

6趋势追踪与质量仪表板

6.1 从 Delta 到 Trend

每个 Delta Record 是微观的"点"，Trend DB 负责把这些点连成"线"——回答管理层面的问题。

趋势指标

指标来源用途
加权风险总分∑(risk_score × weight) 每周趋势判断整体质量方向
Critical 缺口数level=critical 的特性数管理层关注的红色警报
测试有效性中位数test_effectiveness 评分分布测试质量是否在改善
缺陷发现率每周新增 defect_record是变好了还是漏洞在增多
测试覆盖率加权值加权（风险越高权重越大）是否优先覆盖了高风险区
Delta 频率每周 delta 记录数变更活跃度（高→需更多关注）

告警规则

  • 🔴 Critical: 新特性 risk ≥ critical 且零测试 → 阻断 PR
  • 🟠 Warning: 加权风险总分连续 2 周上升 → 发送告警
  • 🟡 Info: 某个特性 30 天未更新风险画像 → 建议重新全量评估
  • 🟢 Good: 测试有效性中位数持续上升 → 质量改善趋势确认

趋势判断逻辑

条件趋势
加权风险 ↓ + 测试有效性 ↑📈 改善
加权风险 → + 测试有效性 →➡️ 稳定
加权风险 ↑ 或 测试有效性 ↓📉 恶化
缺陷发现率 ↑ + 测试有效性 →⚠️ 测试盲区扩大

8总结

🎯 增量评估的核心设计

  • 依赖图映射: 文件 → 特性，变更影响自动定位
  • 指纹缓存: O(1) 跳过未变特性，避免无效重算
  • 变更分类: 7 种变更类型 → 7 种增量策略，计算量分级
  • Delta 输出: 不输出全量，只输出变化量
  • 三层存储: Baseline + Delta + Trend，各自独立演进

📊 三种运行模式

  • PR Check: 增量 · <30s · 阻断高风险零测试合入
  • Scheduled: 全量 · 每周 · 基线刷新 + 趋势更新
  • Manual: 深度 · 按需 · 专项审计 + 新特性评估

🔄 可持续性保证

  • dependency_map 自动维护，不增加人工负担
  • fingerprint 缓存避免重复计算
  • defect_db 增量追加 git log，不重复扫描
  • 趋势自动聚合，月度报告一键生成
  • 告警规则可配置，框架本身也持续演进

🚀 与已有两份报告的衔接

本增量架构设计是前两份报告的工程化落地层：
① npu_test_quality_analysis.html（覆盖审计）→ 提供 baseline features.json 和 tests.json
② npu_risk_based_testing.html（风险策略）→ 提供 risk_profiles.json 和 test_strategies.json
③ 本文档（增量架构）→ 让 ① 和 ② 从"一次性报告"变成"持续运行的工程质量系统"
