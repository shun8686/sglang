SGLang NPU 基于风险的测试策略

从"测没测"到"哪里最可能出问题，出了事有多大影响"——测试资源的最优分配

🎯 核心理念: Risk = Probability × Impact 🧬 4 层测试深度: 基础 → 边界 → 精度 → Fuzz 📊 六大风险因子 + 四大影响因子 🤖 Agent 驱动: Bug考古 → 代码评分 → 策略生成

1为什么覆盖分析不够

覆盖分析能告诉你的

  • ✅ "FuseEP 没有测试"
  • ✅ "W8A8 量化有 8 个测试"
  • ✅ "单元测试数量为 0"
  • ✅ "108 个文件都是端到端测试"
  • ✅ "85% 是 GSM8K 精度阈值测试"

核心问题: 覆盖分析把 18 个特性等权看待，隐含假设"所有代码路径的出问题概率相同"——这显然是错的。

但它不会告诉你

  • ❌ 哪段代码最危险？
  • ❌ FuseEP 的 token dispatch 比 MLA attention 危险 4 倍
  • ❌ 102 个"已有测试"中的 80 个几乎无发现能力
  • ❌ FRACTAL_NZ 格式转换导致的静默错误优先级远高于吞吐下降
  • ❌ 过去 6 个月的 23 个 NPU 修复 PR 中有 18 个集中在 4 个模块

⚠️ 覆盖幻觉

"有测试" ≠ "测试有效"。精度阈值 0.3 的 GSM8K 测试即使通过，也完全可能在下一个 token 上产生错误输出。覆盖数字创造虚假的安全感。

测试的本质是投资决策

测试资源（人力、CI 时间、NPU 算力）有限。每多写一个测试就意味着少写另一个。基于风险的测试回答的是：在有限的资源下，哪些测试的 ROI 最高？

维度覆盖驱动风险驱动
核心问题"测没测？""哪里该多测？哪里可以少测？"

2NPU 风险分类体系

2.1 NPU 专有风险维度

NPU 平台的测试风险不能套用通用分类。需要在三个方面与 CUDA 做差异化分析：

🔢正确性风险

  • 数值精度 — FP32→BF16→FP32 往返误差累积
  • 内存布局 — FRACTAL_NZ vs 标准 NCHW 格式不匹配
  • 量化误差 — dynamic_quant scale 在不同 batch size 下漂移
  • 图捕获 — NPU Graph 重放时的 shape 不匹配
  • Attention — Ascend mask 格式 (上三角) vs CUDA (下三角) 语义颠倒

⚡性能风险

  • 算子融合 — FuseEP 的 fused_dispatch+combine 产生额外内存拷贝
  • 流并行度 — dual-stream MoE 的 stream 同步点成为瓶颈
  • 内存碎片 — NPU KV Cache allocator 在长序列下碎片化
  • CMO 失效 — Cache Management Operation 预取在特定访问模式下退化为顺序读
  • HCCL 拥塞 — all_reduce 在高 TP 下竞争带宽

🔗兼容性风险

  • CANN 版本 — 8.3.RC1 → 8.5.0 的 API 行为变更
  • torch_npu — 新版本引入的算子精度回归
  • HCCL vs NCCL — 集合通信语义差异（如 barrier 行为）
  • sgl_kernel_npu — 独立仓库的版本耦合
  • 模型结构升级 — Qwen3.5 新架构未在旧 CANN 上验证

2.2 风险 = 概率因子 × 影响因子，逐层下钻

不是简单的 P×I 二维矩阵。每个因子都有 NPU 语境下的具体含义和量化方式。

概率因子（6 维）

因子含义NPU 量化方式
代码复杂度圈复杂度、分支数、状态机深度AST 静态分析 + 人工标注
变更频率近期 commit 热点git log --since="90d" | wc -l
历史缺陷密度同类模块的历史 bug 聚类git log --grep="fix\|bug" 关键词匹配
平台差异性NPU 实现与 CUDA 参照的偏离度代码 diff 行数 / 专有 API 调用密度
依赖链深度调用栈中的不可控因素层数import 链分析 (sgl_kernel_npu → CANN → 固件)
测试有效性现有测试的实际发现能力断言类型 × 覆盖率 × 阈值严格度

影响因子（4 维）

因子含义NPU 量化方式
用户暴露面触发该代码的请求占比代码路径热度 (100% 请求 vs 特定模型)
故障模式崩溃 vs 静默错误 vs 性能退化错误传播分析 (crash → NaN → wrong_result → perf)
可检测性用户能否自行发现异常输出偏差的可观测性 (token 分布变化 vs 随机输出)
恢复成本从故障恢复到正常的时间是否需要重启/重载/回滚/集群重建

⚠️ NPU 最危险的风险模式: 静默错误

在 NPU 上，崩溃不是最可怕的，静默错误才是。FRACTAL_NZ 格式转换失败时不会报错——它只是让 tensor 中的值悄悄变化。用户看到的是"模型变笨了"——吞吐正常、延迟正常、没有错误日志，只有输出质量在不可感知地下降。这类故障的 可检测性评分 = 5 (极难检测)，是最需要测试重点覆盖的场景。

4分层测试深度——不同风险，不同策略

风险驱动测试的核心原则：不是"加一个测试"，而是"匹配正确的测试深度"。高风险特性需要多层防御。

🧬 L0 深度验证

Critical 特性

① 数值 Oracle — 已知输入 → 精确输出，逐元素对比
② Metamorphic Testing — 变换输入，验证输出遵循语义不变性
③ 跨平台差分 — NPU vs CUDA 输出逐 token 对比
④ 边界 Fuzzing — 随机组合极端参数，验证不崩溃+输出合法
⑤ 性能回归基线 — 固定配置下的延迟/吞吐 CI 自动检测

FuseEP · Disagg · DynQuantROI: 最高

🔬 L1 多层验证

High 特性

① 正确性 Oracle — HF 参考输出对比
② 边界条件 — min/max/near-boundary 参数组合
③ 性能回归基线 — CI 检测 >10% 退化

DualStream · FRACTAL_NZ · GraphROI: 高

✅ L2 功能验证

Medium 特性

① 多配置冒烟 — 3+ 模型 × 2+ 配置组合
② 合理精度阈值 — 基于历史波动设定 (e.g. 0.8±0.05)
③ 基本功能断言 — 不崩溃 + 输出合法

MLAPO · HiCache · LoRAROI: 中

⚡ L3 基础冒烟

Low 特性

① 单配置不崩溃
② 吞吐不低于基线

基本TP · VLM · SamplingROI: 低

为什么需要分层？

❌ 统一策略的浪费

对"基本 TP 推理"做 Fuzz 测试是浪费——它的代码路径基本是成熟的社区模型代码 + 标准 PyTorch 调用，历史 bug 极少。多出来的精力应该投给 Dynamic Quant。

✅ 分层 = 帕累托最优

对 FuseEP 做 L0 深度验证（5种互补方法）投入 3 人天，预期捕获 80% 潜在故障。同样 3 人天如果平均分配给 18 个特性各 2 小时，什么像样的测试都写不出来。

测试策略（5 层，按优先级排序）

P0 · 数值 Oracle 测试

构造已知分布的 token_per_expert 输入，在 NPU 上运行 dispatch+combine，对比 golden output（手动计算或 CUDA reference）。

投入 1h · ROI 极高 · 可捕获 60% 潜在故障

P0 · 跨平台差分测试

在相同输入下，运行 NPU FuseEP 和 CUDA flashinfer MoE dispatch，逐 token 对比路由结果。差异 >0 即为 bug。

投入 2h · ROI 极高 · 可捕获 CUDA-NPU 移植差异

P1 · 边界 Fuzzing

Fuzz token 数量分布 [0, 1, max_capacity, max_capacity+1, 随机(0, max)] × expert 数量 [1, 8, 64, 256] × top_k [1, 2, 4, 8]。验证不崩溃 + 路由结果合法 + expert 不重复。

投入 3h · ROI 高 · 边界条件是精度 bug 最常见来源

P1 · 数值稳定性测试

对动态量化路径（W4A8, W8A8），验证 quant → dequant roundtrip 在所有 shape 组合下的误差 < 1e-3（相对于 FP32 基准）。

投入 2h · ROI 中-高 · 量化误差累积会导致下游 attention 完全错误

P2 · 压力测试

极端负载：expert_count=256, tokens_per_expert=4096, 16 卡 A3 集群。验证不 OOM、不超时（< 30s）、路由正确。

投入 4h · ROI 中 · 极端场景需特定硬件，但故障影响大

P2 · 性能回归基线

固定配置 (QServe 8B MoE, tp=8, bs=32) 下记录 token/s 基线。CI 自动检测 >10% 退化并告警。

投入 2h · ROI 中 · 性能比正确性次要，但长期有价值

📊 预期效果

以上 6 项测试总投入约 14 人时（~2 人天），预期可将 FuseEP 模块的未检测故障概率降低 75-85%。其中 P0 两项（3h 投入）就能捕获约 60% 的潜在问题——极高的 ROI。

6Agent 在风险测试中的四重角色

从覆盖分析的"文件扫描器"升级为风险分析的"语义分析器"，Agent 承担四个关键角色：

🕵️Role 1: Bug 考古学家

扫描 git log 中 NPU 相关的修复 commit，聚类高频故障模式，关联到具体代码模块。

输入: git log --grep="npu\|ascend\|NPU\|CANN" --since="2025-01-01"
输出: "fused_moe_method_npu.py 过去 6 个月 8 次精度修复
         → 5 次与 dynamic_quant scale 相关
         → 建议做数值稳定性专项测试"

🔍Role 2: 代码风险评分器

基于预定义风险模式库，对每个模块打分，生成风险热力图。

风险模式库（部分）:
+3 手动内存管理 (FRACTAL_NZ, scatter_nd)
+2 多流并发 (stream 同步点)
+2 位运算/整数混用 (w4a4 quant)
+1 NPU 专有 API (torch_npu.npu_*)

🧠Role 3: 隐式风险推断器

识别"代码看起来简单但语义上危险"的模式——Agent 比脚本强在这里。

ascend_backend.py L1540: CUDA mask → Ascend mask 转换
CUDA: [bs, 1, seq, seq] 下三角
Ascend: [bs, seq, seq] 上三角 ⚠️ 语义颠倒
→ 推断: 边界 bs>1 时极易出错 → 需要交叉维度测试

🔬Role 4: 差分风险分析器

对比 NPU 实现与 CUDA 参照，识别专有路径的额外风险。

CUDA: flashinfer.prefill_with_paged_kv_cache(...) ← 成熟 OSS
NPU: 自实现 prefill + 自定义 block table + FRACTAL_NZ 转换
→ 差异: NPU 每条路径都是"自己写的"，缺乏社区验证
→ 策略: 需要比 CUDA 更严格的正确性验证

🤖 Agent 的独特优势

传统脚本能做 Role 1（git log grep）和 Role 2（正则匹配），但 Role 3（隐式风险推断）和 Role 4（跨平台语义对比）是 LLM Agent 的核心价值所在——它需要理解代码的语义意图，而不仅仅是文本模式。

📊 核心结论

同样的 10 人天投入下，风险驱动策略的风险降低效果是覆盖驱动策略的 5-8 倍（50-65% vs ~10%）。原因很直观：把仅有的测试预算投给了"最可能出问题且出了事影响最大"的少数特性，而不是平均分配给所有特性。

9总结与建议

✅ 核心主张

测试不是追求覆盖率数字——而是用最少的资源捕获最多的风险。NPU 平台的测试策略应该从"覆盖驱动"转向"风险驱动"：

  1. 分层测试深度 — Critical 做 L0 深度验证（5种互补方法），Low 做 L3 基础冒烟
  2. 帕累托分配 — 80% 测试资源投给 top 4 风险特性（仅占特性总数的 22%）
  3. ROI 驱动 — 每个测试建议附带预期投入和风险降低估算
  4. 重点防范静默错误 — NPU 上最危险的故障模式是无声无息的精度丢失

⚠️ 关键风险

  1. Agent 的风险评分需要校准 — 初始评分基于代码模式推断，需要人工 review 和持续迭代
  2. 历史缺陷数据可能不完整 — 某些 bug 的 fix commit 未标记 "npu" 关键词，会被遗漏
