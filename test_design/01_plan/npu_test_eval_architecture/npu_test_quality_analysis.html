1.3 测试模式分析

当前 NPU 测试几乎全部采用以下三种模式之一：

模式占比特征质量评价
模式A: GSM8K 精度+吞吐 ~80% 启动服务器 → 跑 GSM8K 200题 → assertGreaterEqual(accuracy, 0.x) 基础冒烟 可发现严重 breakage，无法定位根因
模式B: 功能正确性对比 ~15% SGLang 输出 vs HF 参考输出，逐 token 对比 + 数值容差 较好 有明确正确性 oracle
模式C: 基础功能验证 ~5% 单次请求 + 简单吞吐断言 弱 仅验证不崩溃

3测试质量维度分析

维度现状缺口描述
单元测试 0 NPU kernel、attention backend、量化方法均缺乏独立 UT
正确性测试 少数 (~10%) 大多数模型测试只有精度阈值（如 accuracy ≥ 0.3），无参考对比基准
压力/鲁棒性测试 0 无 OOM 恢复、并发极限、长序列极限测试
回归测试 弱 精度阈值过宽（0.3），难以捕获精度退化
组合测试 0 量化×TP、HiCache×MLA、EP×量化 等交叉维度未覆盖
错误路径测试 0 无 negative test（错误输入、异常配置、资源耗尽）
Mock/轻量测试 0 无 mock attention backend、无轻量验证机制，所有测试依赖真实模型下载

⚠️ 核心发现

108 个测试文件全部是端到端集成测试——每个都启动真实 SGLang 服务器、下载模型、运行评估。这种"全有或全无"的测试策略意味着：

① 反馈周期长（单测 60-400s）
② 依赖网络和模型可用性
③ 难以覆盖边界条件和错误路径
④ 精度退化难以检测（阈值过宽）

4Agent 辅助方案架构设计

4.1 整体架构

4.2 Phase 详情

Phase 1: 代码特性提取

  • 6 个并行 Agent 扫描不同目录
  • 输出 JSON Schema 结构化数据
  • 覆盖: 推理引擎/分布式/量化/Attention/高级特性/接口
  • 深度阅读关键源文件以理解语义
并行执行 ~60s

Phase 2: 测试覆盖映射

  • 5 个并行 Agent 扫描测试目录
  • 提取: 测试类型/断言方式/质量评分
  • 覆盖: 全部 108 个测试文件
  • 自动分类测试模式 (A/B/C)
并行执行 ~60s

Phase 3: 差距分析

  • Pipeline 模式: 每条特性独立流经 3 stage
  • Stage 1: 缺口检测 (匹配特性↔测试)
  • Stage 2: 严重度评级 (4维度评分)
  • Stage 3: 建议生成 (仅 critical/high)
Pipeline ~90s

Phase 4: 综合报告

  • 1 个 Synthesis Agent
  • 整合 Phase 1-3 的所有结构化数据
  • 生成人类可读的 Markdown 报告
  • 含执行摘要/覆盖矩阵/建议清单/统计仪表板
单 Agent ~30s

4.4 关键设计决策

决策选择理由
编排方式 1 个顶层 Workflow phase 进度可视化、token 预算追踪、脚本化可重复运行
Phase 1+2 模式 parallel() 并行 fan-out Agent 扫描范围互不重叠，可完全并行
Phase 3 模式 pipeline() 链式处理 每条特性独立流经 3 个 stage，无需 barrier 等待其他特性
Agent 读取深度 分层: 深度/浅层/不读 Phase 1 深度读代码、Phase 2 浅层扫描测试、Phase 3 纯数据转换
结构化输出 全部 JSON Schema 约束 保证下游 machine-readable，避免解析歧义
建议生成验证 3 视角对抗审查 可行性/唯一性/覆盖率三个 lens 交叉验证建议质量

5Workflow 脚本设计

5.1 核心代码路径

export const meta = {
  name: 'npu-test-quality-analysis',
  description: 'Analyze NPU test quality and generate improvement suggestions',
  phases: [
    { title: 'Extract Features', detail: '6 parallel agents scanning NPU source' },
    { title: 'Map Coverage', detail: '5 parallel agents scanning test files' },
    { title: 'Analyze Gaps', detail: 'Pipeline: gap→severity→suggestion per feature' },
    { title: 'Synthesize Report', detail: 'Final comprehensive markdown report' },
  ],
}

// ====== Phase 1: 并行提取特性 ======
phase('Extract Features')
const features = await parallel([
  () => agent(FEATURE_EXTRACT_INFERENCE,   {label:'feat:inference',  schema:FEATURE_SCHEMA}),
  () => agent(FEATURE_EXTRACT_DISTRIBUTED, {label:'feat:distributed',schema:FEATURE_SCHEMA}),
  () => agent(FEATURE_EXTRACT_QUANT,       {label:'feat:quant',      schema:FEATURE_SCHEMA}),
  () => agent(FEATURE_EXTRACT_ATTN,        {label:'feat:attention',  schema:FEATURE_SCHEMA}),
  () => agent(FEATURE_EXTRACT_ADVANCED,    {label:'feat:advanced',   schema:FEATURE_SCHEMA}),
  () => agent(FEATURE_EXTRACT_INTERFACE,   {label:'feat:interface',  schema:FEATURE_SCHEMA}),
])

// ====== Phase 2: 并行映射测试覆盖 ======
phase('Map Coverage')
const tests = await parallel([
  () => agent(TEST_SCAN_BASIC,     {label:'test:basic_func',  schema:TEST_SCHEMA}),
  () => agent(TEST_SCAN_MODELS,    {label:'test:models',      schema:TEST_SCHEMA}),
  () => agent(TEST_SCAN_INTERFACE, {label:'test:interface',   schema:TEST_SCHEMA}),
  () => agent(TEST_SCAN_CI,        {label:'test:ci_workflows',schema:TEST_SCHEMA}),
  () => agent(TEST_SCAN_KERNEL,    {label:'test:sgl_kernel',  schema:TEST_SCHEMA}),
])

// ====== Phase 3: Pipeline 分析 ======
phase('Analyze Gaps')
const allFeatures = features.filter(Boolean).flatMap(f => f.items)
const allTests = tests.filter(Boolean).flatMap(t => t.tests)

const results = await pipeline(
  allFeatures,
  // Stage 1: 缺口检测
  async (feature) => {
    const gap = await agent(GAP_DETECT_PROMPT(feature, allTests), {
      label: `gap:${feature.name}`,
      schema: GAP_SCHEMA
    })
    if (!gap.has_test) {
      // Stage 2: 严重度评级
      const severity = await agent(SEVERITY_PROMPT(gap), {
        label: `severity:${feature.name}`,
        schema: SEVERITY_SCHEMA
      })
      // Stage 3: 建议生成 (仅 critical/high)
      if (severity.level === 'critical' || severity.level === 'high') {
        const suggestion = await agent(SUGGEST_PROMPT(gap, severity), {
          label: `suggest:${feature.name}`,
          schema: SUGGESTION_SCHEMA
        })
        return { ...gap, ...severity, suggestions: suggestion.items }
      }
      return { ...gap, ...severity, suggestions: [] }
    }
    return { feature: feature.name, has_test: true, severity: null }
  }
)

// ====== Phase 4: 综合报告 ======
phase('Synthesize Report')
const report = await agent(SYNTHESIS_PROMPT(allFeatures, allTests, results))

return { features: allFeatures, tests: allTests, gaps: results, report }

6.3 风险矩阵

风险概率影响缓解措施
特性遗漏（is_npu() 隐式分支） 中 中 Phase 1 Agent 增加专项 prompt: "搜索所有 is_npu() 条件分支"
Agent 特性理解偏差 低 中 JSON Schema 约束输出格式 + 人工抽查 20% 结果
测试分类错误 中 低 quality_score 由明确标准判定（是否有参考对比？是否检查数值容差？）
建议不可落地 中 高 每个建议带 reference 字段指向 CUDA 同类测试；人工 review 确认
模型/网络依赖导致 Agent 失败 低 高 Agent 只读本地文件，不启动服务器、不下载模型

📊 总结

Agent 辅助方案完全可行。核心价值：

① 速度 — 3-5 分钟完成人工 2-3 天的工作量
② 系统性 — 不会因疲劳或疏忽遗漏特性
③ 结构化 — JSON Schema 保证输出可直接用于下游工具（CI 配置、测试骨架生成）
④ 可重复 — 脚本化后可持续追踪测试债务的收敛

最大风险是建议质量——Agent 对 NPU 硬件特性的理解有限，生成的测试建议需要人工确认可行性和优先级。
